PatchRelay stack

Tools that ship through GitHub truth.

PatchRelay writes branches, review-quill reviews changes, and merge-steward lands tested integration trees. The services stay separate; GitHub is the bus.

The cast

One PR, three independent roles.

Author

PatchRelay owns Linear delegation, worktrees, implementation, review fixes, CI repair, and queue repair.

Reviewer

review-quill reads GitHub PR truth and publishes ordinary approve or request-changes reviews.

Lander

merge-steward admits approved green PRs, builds speculative integration branches, validates, and fast-forwards main.

Shared bus

They do not call each other.

push branch-> GitHub PR-> review-> checks-> queue-> main

The protocol boundary is GitHub: PR head, review state, check state, queue check runs, and spec branches.

Mental model

A PR is a ref. A change is a patch.

headSha

The exact commit GitHub currently calls the PR head. Reviews are attached to this world.

patch_id

The stable identity of the diff. Rebases and equivalent force-pushes can keep the same change identity.

git diff $(git merge-base <base> <head>)..<head> | git patch-id --stable
Integration truth

The queue tests what will land.

tree   = git merge-tree --write-tree main pr-head
commit = git commit-tree $tree -p main -p pr-head
push   mq-spec-N
test   mq-spec-N
green  -> fast-forward main

Branch CI tells you the PR head works in isolation. Spec CI tells you the integrated tree works on today's main.

The key invariant

Requested changes require a new head.

After review-quill requests changes at SHA A, PatchRelay must push SHA B before the issue can return to review or queue.

Blocked

same head Review fix finishes, but GitHub still reports SHA A. PatchRelay fails the run.

Allowed

fresh head Review fix publishes SHA B. The reviewer can now evaluate new facts.

Production evidence

The guard fired in the real system.

Requested-changes run finished for PR #355 without pushing a new head
past blocking review SHA 7586be6a;
PatchRelay must not hand the same SHA back to review.

That is a protected stop. It means PatchRelay refused to ask the reviewer to reconsider an unchanged PR head.

Operator map

When something stalls, follow the owner.

In Progress

PatchRelay owns the worktree. Use patchrelay status ISSUE to observe it or issue open to take over.

In Review

Look at PR reviews and required checks. Red branch CI can trigger ci_repair.

In Deploy

Look at merge-steward. Branch CI is metadata; the eviction check is the repair signal.

Queue repair

Eviction is a handoff, not a mystery.

spec fails-> merge-steward/queue-> queue_repair-> fresh head-> approval-> merge 
merge-steward queue show --pr <num>
patchrelay status APP-123
patchrelay logs APP-123 --lines 100
The whole thing

The tools are small because the protocol is small.

  1. PatchRelay changes the branch.
  2. review-quill reviews the change identity.
  3. merge-steward tests the integration tree.
  4. GitHub carries all signals between them.
  5. A repair always returns with a fresh head.
